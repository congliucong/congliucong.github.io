Java基础之LongAdder

发表于 2020-06-27 分类于 Java基础 Valine：
本文字数： 5.8k 阅读时长 ≈ 5 分钟

最近在翻看ConcurrentHashMap源码计数部分时，发现counterCell的实现原理其实跟LongAdder分段技术原理类似，因此深入学习一下LongAdder的实现原理。

AtomicLong

为什么要先记录AtomicLong，因为只有对比，才能更加理解原理之间的关系。

我们都知道AtomicLong原子类是依赖底层Unsafe类的CAS来保证原子性。CAS即比较替换，只有当内存里的值等于预期值时，才会使用原子的方式把内存的值修改为更新值，整个操作是原子性的。代码层面是有while循环不断循环重试，直到成功，底层方面是通过cpu的cmpxchgl指令来实现的。

扯远了，那为什么Java8还要推出LongAdder呢。

因为虽然大部分情况下，CAS都能提供不错的性能，但是在多线程高竞争的情况下，不断自旋重试这个操作可能带来成倍的开销。因此，在这种情况下推出LongAdder。

LongAdder

原理阐述

那为啥LongAdder在高并发下的性能会比AtomcLong更好呢。

这是AtomicLong里，高并发情况下多个线程竞争的是一个由volatile修饰的vaule。所有操作都对这一个值进行竞争，也就是说这个变量是一个热点。

但是在LongAdder的设计思想中，将这一个热点分散，将一个value分散到一个数组中，不同线程命中到数组的不同槽里面，这样不同线程只选要对自己命中的值进行CAS操作即可，很好的分散了热点，冲突的概率就小了很多。

其实这个思想在ConcurrentHashMap中多次用到，JDK1.7的数据结构分段锁segment。JDK1.8时增加concurrentHashMap的size时。同样的，这也是一种空间换时间的思想。

因此，在一般业务下使用AtomicLong即可，如果并发量很大，且写多读少的情况下，可以选择LongAdder。

源码解读

LongAdder继承自Striped64。

而Striped64内部维护了一个懒加载的数组以及一个额外的base值，数组的大小是2的N次方，使用每个线程Thread内部的的哈希值来访问。

数组cells的Cell类由@sun.misc.Contended来修饰。这个我们在ConcurrentHashMap的countCell中提到过，是用来解决伪共享的问题。

解决伪共享的方法是字节填充。

add方法

当执行到longAccumulate方法时，说明出现了竞争。

所以，总的来说，大致流程是，在无竞争的情况下直接更新base域。在第一次发生竞争时，即CAS失败的时候，会创建一个大小为2的cells数组。每次扩容都会加倍，直到达到cpu核数。扩容的时候只能由一个线程来执行，因此需要一把锁。而在LongAdder中，巧妙的通过CAS更新cellsBusy来实现一个简单的自旋锁。

如何确定数组索引？

而数组访问索引决定了当前线程将值累加到cells数组的哪个位置，实现方式是通过Threa类中的threadLocalRandomProbe域取模实现。而threadLocalRandomProbe域的更新，则是由ThreadLocalRandom来进行更新的。

为什么要cells数组大小要限制cpu核数？

因为即使有超过cpu核数的线程去更新，但是每个线程也只会跟一个cpu绑定，更新的时候最多也只有cpu核数个数的线程在同时操作，因此只需要通过hash将不同线程的的操作分散到不同的槽位上即可。

总结

因此，基于这种分散热点的设计，在高并发时将对单一变量的cas操作分散到数组cells中多个元素的cas操作，取值时再进行求和，并发较低时则直接对base变量进行cas操作。因此，LongAdder从性能角度来说是比AtomicLong优秀很多。

但是在使用角度，AtomicLong的一系列方法不仅可以自增，而且可以返回更新后的值，而LongAdder的increment、decrement方法均为void方法，如果需要获取自增后的值还是AtomicLong比较方便。

所以，可以基于业务场景进行判断。
