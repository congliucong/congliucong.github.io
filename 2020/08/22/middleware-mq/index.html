MQ常见问题

发表于 2020-08-22 分类于 消息中间件 Valine：
本文字数： 2.1k 阅读时长 ≈ 2 分钟

消息中间件是我们项目开发中使用最频繁的一个中间组件。

这里稍微对一些常见的问题进行一下总结，后续有时间再研究一下源码。

为什么要使用消息中间件？

  1. 解耦。

解耦最大的好处就是减少系统复杂度。

项目中实际用到的场景有：支付下单后发送MQ生成履约信息、库存扣减后发送消息更改订单状态、发货完成后发送MQ生成配送单等等。

  1. 异步

使用MQ可以将一个复杂的请求分割成多个异步的请求，主流程完成之后即可返回信息，提高用户体验。如果ABCD四个业务流程，每个流程耗时100ms，如果不进行异步处理，则总耗时400ms，使用MQ之后便可以将A完成后发送三个MQ信息到队列里面，即可返回结果。这样对于用户来说是透明的。

  1. 削峰

在高并发高请求的场景下，如果系统直接基于数据库，很容易把数据库搞崩。使用MQ之后，可以将请求平稳的进行处理。

当然，引入MQ也同样增加系统的复杂度。这个下面再讲。

MQ的技术选型

  1. ActiveMQ，单体吞吐量万级，比RocketMQ、Kafka低一个数量级。
  2. RabbitMQ，单机吞吐量万级，最大特点是开发语言是erlang，导致后续扩展不好扩展。
  3. RocketMQ，10万级吞吐量，分布式架构、参数优化配置可以做到0丢失、并且是分布式的。
  4. Kafka，高吞吐量，一般配合大数据系统实施计算场景。

如何保证MQ高可用？

以RocketMQ为例:

  1. 单个master模式
  2. 多个master模式
  3. 多个master多slave模式-异步复制
  4. 多个master多slave模式-同步双写

对于RabbitMQ来说，一般可以部署为镜像集群模式实现高可用性。在镜像集群模式下，创建的queue，无论元数据还是queue里的消息都会存在多个实例上。

如何保证消息不被重复消费？

消费端保证系统幂等。

根据业务场景来，比如：

  1. 如果是写库，那么先拿主键查一下，有就不消费了。
  2. Redis存储已经消费的唯一主键ID
  3. 基于数据库的唯一主键约束等等

如何保证消息不丢失？

以RocketMQ为例：

  1. 对于生产者来说，
    • 默认情况下采用同步阻塞方式发送，检查发送状态，如果是OK，则表示消息一定成功投递；
    • 采用事务消息投递方式，如果消息发送失败，消息会存储在commitLog里面。
  2. 对于Broker来说，
    • 消息支持持久化到commitLog里面，即便宕机重启，未消费的消息也是可以加载出来的。
    • broker自身支持同步刷盘、异步刷盘的策略，能够保证接收到的消息一定存储在本地内存
    • broker集群支持1主多从策略，支持同步复制异步复制，同步复制即便mater崩溃，消息也不会丢失。
  3. 对于消费者来说，
    • 消费者自身维护一个持久化的offset，标记已经消费或者已经成功发回borker的消息下标。
    • 消费者消费失败，会把这个消息发挥broker，发回成功后，更新自己的offset。
    • 消费者消费失败，发回broker，如果broker挂掉，那么消费者也会定时重试这个操作。
    • 如果消费者和broker一起挂，消息也不会丢失，因为消费者里面的offset是定时持久化的，重启之后，继续拉取offset之前的消息到本地。

如何保证消息的顺序性

如果MQ组件不能保证消息顺序，那么 需要我们去保证，首先要保证相同的消息比如相同订单的消息进入同一个队列里面，其次生产者要按顺序将消息投递到队列里面，只能有一个消费者只能单线程中逐个消费消息，这样能够保证消息的顺序性。

RockMQ支持顺序消息。实现原理是将不同订单的消息路由到不同分区中，消费者消费时通过一个分区只能有一个线程消费的方式保证消息顺序。

RocketMQ的重要组件

主要包括NameServer、Broker、Producer、Consumer。

  1. NameServer是一个简单的topic路由注册中心。类似dubbo的zk，支持broker的动态注册和发现。同样也是集群部署，每个实例之间不通信，但是每个实例上都保存了所有的信息。主要功能是：
    • Broker管理，接受Broker集群注册信息并保存下来，提供心跳机制检查Broker是否还存活。
    • 路由信息管理，保存关于Borker集群的整个路由信息和用于客户端查询的队列信息。
  2. Broker，主要负责消息存储，投递和查询以及服务高可用。
  3. Producer，消费生产者，支持分布式集群部署，通过负载均衡模块选择相应的broker中的queue进行投递。
  4. Coumser，消息消费者。

RocketMQ的消息有哪些类型？

  1. 普通消息
    • 同步消息，一条消息发送成功并且接收到返回消息才能发送下一条消息
    • 异步消息，一条消息发送后接着发送下一条，消息异步返回
    • 单向发送，一条消息发送后不需要反馈就继续发送消息。
  2. 顺序消息
  3. 事务消息
  4. 延时消息

如果让你设计一个MQ，你应该怎么设计？

首先MQ要支持可用性。注册中心集群部署，多副本，leader–follower模式，broker挂了重新选举leader对外服务。

保证MQ的数据的持久性，数据落盘，同步刷新，顺序写，减少磁盘随机读写的寻址开销。

保证MQ的可伸缩性，快速扩容，每个topic下再细分，每个partiton一台机器，存放一部分数据，资源不够，扩容，给topic下增加partition，然后进行数据迁移，提高吞吐量。

保证下游消费者消费消息不丢失，必须要求有返回消费状态信息等等。
