Spring之常见问题总结

发表于 2020-08-02 分类于 Spring Valine：
本文字数： 3.4k 阅读时长 ≈ 3 分钟

Spring源码学习是真的一不看就忘，然后就只能记个大概。

这里对前面几篇大概分析过的流程进行一下总结。

加油ヾ(◍°∇°◍)ﾉﾞ

Spring里面用到了哪些设计模式？

  1. 单例模式。 Spring中的bean都默认是单例的。
  2. 工厂模式。 最经典的就是BeanFactory和ApplicationContext来生产Bean对象。
  3. 代理模式。 最常见的就是AOP的实现方式就是通过代理来实现的。Srping中主要是使用JDK动态代理和CGLIB代理。
  4. 适配器模式：在AOP中，利用适配器模式，对于不同advisor，根据不同的适配器包装成各MethodInterceptor。
  5. 模板方法模式：例如数据库相关的类中，对于数据库的建立连接、执行查询、关闭连接等过程，都使用与模板方法。

谈谈对IOC的理解以及它的实现原理？

IOC，控制反转，主要是通过Spring来管理对象的创建、配置和声明周期。

实现原理，从源码角度主要分为两个部分。

  1. IOC容器初始化过程。
  2. 加载bean过程。

IOC容器初始化阶段主要是：

  1. xml文件或者注解由资源定位器ResourLoad进行加载为抽象的资源Rourse
  2. 当初始化BeanDefinitionReader时，定义ResourceLoader，当执行loadBeanDefinition方法时，会将Rource根据id、name属性等解析构成成BeanDefinition，经过BeanDefinitionHolder包装后返回。
  3. 最后由BeanDefinitionRegister对BeanDefinitionHolder的bd进行注册，IOC容器是通过一个ConcurrentHashMap来维护Bd的，注册的过程其实就是将Bd放入到这个map里面，key是BeanName，value是Bb。

当IOC容器初始化完毕之后，当显示调用BeanFactory.getBean方法时，才开始Bean的实例化过程。这个过程主要流程是

  1. doGetBean方法实现的，首先尝试从getSinleton()方法里面获取bean，这个方法主要是从三级缓存中取bean。
  2. 如果没取到，则获取BeanDefinition信息，进行校验，然后根据单例还是原型执行不同方法，以单例为例，执行getSingleton方法，回调createBean方法。在实例化之前会给BeanPostProcessor返回一个代理对象的机会。
  3. 随后执行doCreateBean方法实例化bean。实例化bean会创建一个BeanWrapper对象，这是BeanDefinition到Bean的一个中间产物
  4. 通过这个BeanWrapper进行属性填充。
  5. 最后进行初始化bean，激活Aware接口，执行BeanPostProcessor前置处理器，判断是否实现InitializingBean，是的话执行afterPropertiesSet方法，判断有没有init-method，最后执行Bpp的后置处理器。这里也就是AOP实现的原理。
  6. 最后就能获取实例化的对象。

谈谈对AOP的理解以及它的实现原理

Aop，面向切面编程。Spring Aop基于动态代理的方式来实现的。如果实现了接口就使用JDK动态代理，反之使用CGLib代理。

JDK动态代理主要针对的对象是类实现了某个接口。底层是基于反射的机制实现的，生成一个同样实现接口的代理类。而CGLIB代理是针对没有实现接口的，底层原理是基于asm第三方框架，通过修改字节码生成子类，重写父类方法，实现代码的增强。

Spring AOP的原理是：

比如最简单的ProxyFactoryBean为例：

  1. 当通过proxyFactoryBean.getObject()时，根据单例还是原型不同类型生成不同的代理对象。

  2. 以单例为例，因为proxyFactoryBean继承ProxyCreatorSupport，所以通过createAopProxy生成AopProxy对象，这个有两个实现类1. cglib 2.jdk。根据目标对象是否有接口来返回不同的实现类，最后通过AopProxy.getProxy来获取代理对象。

  3. 以JdkDynamicAopProxy为例，获取到代理对象后，当调用代理对象的方法时，都会转发到aopProxy.invoke方法上。

    1. 首先判断是否在线程内部暴露

    2. 其次，获取该房费的拦截链条，通过不同的advisor，使用适配器模式包装成不同的methodInterceptor.

    3. 如果拦截链为空，则直接执行proceed方法。如果不为空，则将拦截链包装为MethodInvocation，执行其proceed方法。

    4. 最后依次执行里面的拦截链，例如after的，就先执行proceed，然后执行afterReturning()，这样就可以将拦截器后移。

并且Spring Aop属于基于动态代理实现的，属于运行时增强。而AspectJ Aop属于编译时增强，也就是支持编译前后和类加载时织入。

FactoryBean和BeanFactory有什么区别？

BeanFactory，以Factory结尾，表示他是一个工厂类，用于管理Bean的一个工厂。是IOC的核心接口，用来实例化bean等。Spring中有很多中实现，例如XmlBeanFactory、ApplicationContext等。

FacotryBean，以bean结尾，表示他是一个bean。与普通bean不同的是，它是一个能生产或者修饰、定制化对象的bean。如果一个对象实现了FactoryBean，通过BeanFactory.getBean获取的对象实际上是FactoryBean.getObject获取到的对象，如果想要获取FactoryBean本身，那么需要在beanName前加个&符号。应用实例包括：Spring里面ProxyFactroyBean、mybatis的SqlSessionFactoryBean等。

SpringBean的生命周期

SpringBean的生命周期大概指的是：

  1. 实例化bean。根据构造函数或者工厂方法初始化bean，并用BeanWrapper包装。
  2. 属性注入 ，将BeanDefinition中的属性赋值给BeanWrapper对象。
  3. 初始化，包括了激活Aware接口、BeanPostPorcessor增强处理器、注册回调函数等这样就能够供使用。
  4. 执行销毁方法。

Spring怎么解决循环依赖的？

  1. Spring对于非单例模式的循环依赖不解决，直接抛出异常。
  2. Spring对于全是构造器的循环依赖也是无法解决的。

所以，Spring对于不是这两种情况的的循环依赖时通过三级缓存来解决的。在doGetBean方法里面，首先会调用getSingleton方法，从三级缓存中取出对象。

一级缓存：singletonObjects 存放的是实例化、初始化完毕的对象。

二级缓存：earlySingletonObjects 存放的是实例化完成，但未初始化完成的对象。

三级缓存： singletonFactories ：存放的是对象工厂。

当发生循环依赖时，即AB相互依赖，当创建A时，将创建过程中的bean的ObjectFactory加入到三级缓存中，如果A被AOP代理，那么通过这个ObjcetFactory获取到的A代理后的对象。如果A没有被Aop代理，那么获取到的就是A实例化后的对象。当A进行属性注入时，会去创建B，同时B又依赖A，所以创建B的同时又会调用getBean(a)来获取需要的依赖，此时的getBean(a)会去缓存中取，第一步先获取三级缓存中的ObjectFactory，随后调用其getObjet方法获取对应的对象，将其注入到B中。紧接着B走完它的生命周期流程。当B创建完后，再注入A，此时A走完生命周期流程，至此循环依赖结束。

一定要用三级缓存吗？二级缓存可以不？

如果使用二级缓存解决循环依赖问题，意味着所有Bean在实例化后就要完成AOP代理。违背了Spring设计的原则，Spring设计之初就是通过AnnotationAwareAspectJAutoProxyCreator这个后置处理器来在生命周期的最后一步完成AOP代理。
