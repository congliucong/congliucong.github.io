中间件之Canal

发表于 2020-08-08 分类于 消息中间件 Valine：
本文字数： 698 阅读时长 ≈ 1 分钟

最近工作中引入了Canal，业务场景是这样的：当履约计划并更时，Canal监听履约计划的变更，从而对对应的发货单进行处理。并且项目中用到的DTS阿里云服务，可能也是基于Canal来实现的，这个有待考证。

所以这里就简单总结一下Canal的原理。

什么是Canal?

官网原话是：Canal是用于Mysql数据库增量日志解析，提供增量数据订阅和消费的一个组件。

工作原理

Mysql主备复制原理，在Mysql章节已经提到过，这里在提一下：

Mysql master将数据变更写入到二进制日志即binlog日志，Mysql slave将master的binlog日志拷贝到它的中继日志中，进行重放，从而实现mysql的主从复制。

所以Canal的原理即基于此：

  • canal模拟Mysql slave的交互协议，伪装自己为Mysql的从节点，向Mysql master发送dump协议。

  • Mysql master收到dump请求后，开始推送binlog日志给canal，随后canal解析binlog日志。

支持模式

canal支持将数据同步:

  • MQ，包括Kafka、RocketMQ
  • Mysql
  • ES

使用Canal中遇到一些什么问题？

canal的消息格式问题。

canal.mq.flatMeaasge 配置是否为json格式。

如果设置为false，那么对应MQ收到的消息为protobuf格式，需要用CanalMessageDeserializer进行解码。

如果为trur，生产到mq的消息就是json，否则就是protobuf二进制。
