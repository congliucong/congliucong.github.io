面试题整理

发表于 2020-05-23 分类于 杂谈 Valine：
本文字数： 2.6k 阅读时长 ≈ 2 分钟

对一些常见或者不常见的面试题记性一些整理。

好记性不如烂笔头。

讲讲你知道的一些锁？

这个问题问范围很广泛，我们可以从多个角度来说。

如果从Java的角度：

  1. 悲观锁、乐观锁。
    1. 悲观锁有 synchronized关键字、以及Lock的实现类比如ReentrantLock、ReentrantReadWriteLock等。
    2. 乐观锁最常见的就是CAS。
  2. 自旋锁、适应性自旋锁。自旋锁的实现原理同样是CAS。而适应性自旋锁即自旋次数不在固定，而是由上一次自旋时间及锁的拥有者状态决定，即刚刚获取过锁，jvm会认为这次自旋很可能再次成功，进而允许自旋等待时间更长，
  3. 对于synchronized关键字还有 无锁、偏向锁、轻量级锁、重量级锁。
  4. 公平锁、非公平锁。公平锁是按照多个线程申请先后来获取锁，吞吐效率会比较低。而非公平锁是多个线程加锁时直接尝试获取锁，获取不到才会到等待队列的队伍等待。ReentranLock里面公平模式、非公平模式。
  5. 可重入锁、非可重入锁。可重入锁又叫递归锁，即获取锁后再次进入会自动获取锁，java中ReentrantLock和Synchronized都属于可重入锁。
  6. 独享锁、共享锁。独享锁又叫排它锁，指的是锁一次只能被一个线程锁拥有。Java中ReentrantReadWriteLock中writeLock属于独占锁，ReadLock属于共享锁。、

如果抛开Java来说，还有

  1. 分布式锁，实现方式有Redis、ZK、甚至Mysql都可以作为分布式锁。
  2. Mysql中有行锁、表锁、gap锁、next-key锁等等。

谈谈你对分布式锁的理解？

现在项目大多不是以前的单机部署时代，在那种情况下，对同步资源使用Jdk自带的锁机制就可以完成。但对于目前的分布式环境下，对共享资源的抢占问题用单机的锁就没有作用了。因此才会引入分布式锁。

所以分布式锁方案大致一般有：

  1. Redis分布式锁。

  2. ZK分布式锁。

  3. 甚至Mysql可以实现分布式锁。

    1. 悲观方案：在Mysql里面搞一张专门存储分布式锁信息的表，其中一个字段为分布式锁的key，并且key是一个唯一索引。当多个请求发起，先进行加锁，也就是多个请求拿着同一个key来进行数据插入操作，数据库可以保证只有一个请求插入成功。

      其他没有获取到锁的请求，根据功能需求：如果需要继续获取锁的，就周期性尝试；如果获取到锁，则方法正常结束。

      当获取锁成功的这个请求处理完毕后，再执行删除语句，对锁进行释放。

      如果还需要可重入，则加两个字段：线程名称、重入次数。

      当然这种方案锁没有失效时间，如果执行删除语句失败后，就是一把死锁。

      补偿方案就是定时任务定期清除超时的数据，防止死锁。

    2. 乐观方案：就是mysql的版本号。操作数据的时候，先拿出数据的版本号+1，执行的时候对比原版本号是否相同。

redis和mongoDB的区别

  • 内存管理机制

    • Redis数据全部放在内存里面，定期写入磁盘，当内存不够时可以指定淘汰策略删除数据
    • MongoDB数据存在内存中，当内存不够时，只将热点数据放入内存，其余数据存在磁盘。

  • 持久化方式

    • redis数据持久化是使用RDB或者AOF方式。
    • MongoDB持久化则把操作的数据通过mmap方式映射到内存某个区域，在这个区域修改数据避免了零碎的硬盘操作。

  • 数据结构

    • Redis支持数据结构丰富：包括string、list、hash、set 、sorted set。
    • MongoDB数据结构比较简单，面向文档存储。

说说Java对象创建过程

  • 当jvm遇到一条new执行时，首先检查这个指令的参数能不能在常量池中定位到一个类的符号引用，并检查这个符号应用代表的类是否被加载、解析、和初始化。如果没有就进行类加载过程。
  • 类加载检查通过后，为新生对象分配内存。分配内存两种方式：指针碰撞或者空闲列表。为了保证分配内存线程安全，采用对分配内存动作同步处理即CAS+失败重试。或者 本地线程分配缓冲TLAB，哪个线程分配内存在哪个线程的缓冲区进行。
  • 随后进行初始化，即初始化为零值，设置对象头等。
  • 最后执行构造函数。

说说你在工作中遇到的难点？

  • 技术方案选型。比如当遇到一个问题的需求或者一个优化点的时候，首先需要调用实现功能的方案，当可能有多种技术方案来实现这个功能，对比方案的优缺点，以及结合业务场景选出最合适的方案。这个过程让人成长比较快。比如：遇到分布式事务时，需要生成分布式ID该怎么做？设计缓存方案时如果做一些预防性措施等。
  • 当有突然业务场景出现时，如何能快速变更程序响应业务。当业务出现变更时，解决完当时的问题，无非是订正数据，订正完了之后，如何将现有业务代码变更后适应新的业务场景。例如：当小店配送范围并更后，怎么快速将用户订单、履约单、发货单、配送单等都进行修改。这个问题出现后，我们当时是先开发一个临时预案订正整个链路的单据，但当这种需求多次出现的时候，就需要将其产品化以及在程序上功能支持，我们当时是引入了Canal，监听上游单据出现变更时，监听Canal发送的MQ消息，随后对整个链路进行变更。
  • 其余，就是一些具体的技术场景。比如：
    • 当更新es数据后，立刻发起查询，查询到的还是修改前的数据。这是因为对ES原理不熟悉的原因造成的，ES是近实时搜索，写入和打开一个新段的轻量的过程叫做refresh，默认情况下每个分片会每秒自动更新一次，所以文档变化不是立即对搜索可见。这种问题的解决办法可以每次调用refresh API手动刷新一次即可，但是这么干很容易把整个集群拖垮。在这种高频场景下更新，应该考虑配合其他方案使用。比如前段来一端动画，或者这种情况直接走数据库。
    • 更新履约计划后，需要重新生成发货单，查询发现，发货单一直没有正确的生成。发货单是全部错乱的，该删的没删没删全，该新增的没新增全或者新生成根本是原来的发货单。慢慢排查，是因为第一个原因是：发生发货单的MQ被消费的时候，原先监听数据库的Canal在原履约计划删除的时候会发送MQ删除发货单，但是这个MQ是一条一条消费的，在没消费完毕时，就新增了新的发货单，导致新增的又被删除。这种情况通过生成发货单的MQ改为延时消息，延时10s等Canal的消息完全消费后再执行新的发货单生成。还有一个原因是修改履约计划后发送发货MQ的是调用以前的接口，这个接口里面是使用多线程的。这就导致修改履约计划的事务可能还没提交，新的线程重新查询履约计划时仍查到原来的记录。解决办法是：在事务提交后执行afterCommit方法里面去调用生成发货的方法。
