Java基础之FutureTask

发表于 2020-07-18 分类于 Java基础 Valine：
本文字数： 17k 阅读时长 ≈ 16 分钟

前篇文章说道，创建线程的第三种方式是实现Callable接口，然后使用FutureTask类来进行包装，因为FutureTask类实现了RunnableFuture接口，并且RunnableFuture接口又继承Runnable和Future接口，因此可以通过FutureTask作为Thread类的构造方法的参数来创建线程。

这篇文章就从源码的角度来分析一下。其实说分析也不对，主要在学习别人优秀的博文，来记录学习一下。

Callable接口

这又是Doug Lea大佬写的，膜拜膜拜~

观察这个接口，可以发现跟Runnable接口类似，但是又有两点不同：

  1. Callable接口的call方法有返回值。
  2. 并且该方法还可以抛出异常。

但是，光使用这个接口是有问题的，首先是直接调用call方法是在当前线程里面调用，其次如果call方法执行较久，那么线程则阻塞在这里，所以肯定有其他机制配合call接口使用，达到效果是：将call方法交给其他线程执行，并在合适的时候判断任务是否完成，并获取线程执行的结果。那就是Future接口。

RunnableFuture接口

因为futureTask类实现了RunnableFuture接口，因此也需要对这个接口进行了解。

看到其实现，很容易看到，RunnableFuture接口继承了Runnable接口和Future接口。所以可以通过Runnable接口来执行线程任务，通过Future接口来获取执行结果或者进行其他操作。

如何将一个Runnable接口转为Callable接口

通过Excutors工具类。

这个方法采用了设计模式中的适配器模式，将一个Runnable类型对象适配成Callable类型，返回的结果是我们传入的result参数，为了适配，感觉没啥意义。

FutureTask

首先明确FutureTask是一个类，实现了RunnableFuture接口，而RunnableFuture接口则继承了Runnable接口和Callable接口。其次，跟AQS类似，FutureTask也有几个核心的概念：状态标志state、队列、cas。

状态state

state属性跟AQS中的state一样，是贯穿整个FutureTask的最核心属性。其实从注释中可以看出来，状态的转移有这几种形态，1个初始状态，2个中间状态，以及4个终止态。

分别是：

  1. 初始状态：NEW
  2. 中间状态：
    • COMPLETING 正在设置任务结果
    • INTERRUPTING 正在中断运行任务的线程
  3. 终止状态：
    • NORMAL：任务正常执行完毕
    • EXCEPTIONAL：任务执行过程中发生异常
    • CANCELLED：任务被取消
    • INTERRUPTED：任务被中断

需要注意的是：中间态非常短暂，不是表示任务正在被执行，而是任务已经执行完毕，正在设置最终的返回结果。

而设置一个任务的状态为终止态，只有三个方法：

  • set
  • setException
  • cancel

队列

在FutureTask中，队列的实现是一个单向链表，表示所有等待任务执行完毕的线程集合。那么其大概工作流程是，线程获取任务执行的结果，如果任务还没执行完毕，那么将其放在等待队列中挂起，直到任务执行完毕后将其唤醒。这其实跟AQS的工作原理类似。

队列节点

结构其实很简单，只有一个线程属性和下一个节点。

FutureTask中的这个单向链表当做栈来使用，并且是一个Treiber栈。并且是通过CAS来完成入栈出栈操作，因为同时可能有多个线程想要获取任务的执行结果，可能出现多个线程同时入栈的情况，因此通过CAS保证入栈的线程安全。

因此，通过一个指向栈顶节点的指针 即可使用这个队列，即waiters属性。

那是怎么通过CAS来保证出入栈的线程安全呢？

答案跟AtomicLong等原子类一样，使用UNSAFE魔法类。

其中runner属性代表的是执行FutureTask中的“Task”的线程。

最后总结run方法：

接下来分析Future接口的几个方法：

cancel(boolean mayInterruptIfRunning)

首先，如果state状态不为NEW，直接返回false，这是因为如果状态不为NEW，说明任务肯定已经执行完毕了。

如果状态为NEW，那么根据mayInterruptIfRunning的值将NEW状态设置为INTERRUPTING或者CANCELLED，接下来执行try里面的逻辑：

如果mayInterruptIfRunning为true，并且runner即当前执行任务的线程不为mull，中断当前正在执行任务的线程，最终将状态置为INTERRUPTED。中断完成之后，执行finishCompletion()来唤醒所有栈中等待的线程。

总结，cancel完成以下两种状态转变：

  1. NEW ->CANCELLED。（当mayInterruptIfRunning属性为false）
  2. NEW -> INTERRUPTING -> INTERRUPTED。（当mayInterruptIfRunning属性为false）

所以，对于第一条路径，虽然最终返回了true，但是也只是将状态置为CANCELLED，并不会中断线程的执行，带来的后果是：任务即使执行完毕了，也无法设置任务的执行结果。因为set()方法里面，设置任务结果有个中间态，是以当前state状态为NEW作为前提的，如下图：

对于第二条路径，中断当前正在执行的线程，但是，在上一篇文章中，我们知道，interrupt()方法只是会清除中断标志位，如果线程处于阻塞状态，则会抛出异常。所以，响不响应这个中断完全由线程自己决定，即我们代码里面实现，即call方法内部是否对中断进行了响应，是否将中断异常抛出。

回到run方法，catch (Throwable ex)是将所有异常都捕获，包括中断异常。

但是，即便进入了catch (Throwable ex)，也不会将异常信息设置为outcome里面，跟上面那个一样，要求状态是NEW，才会设置异常。

所以，setException(ex)这个方法是处理任务自己执行过程中的异常，如果我们没有主动去cancel，那么状态一直为NEW，如果发生了异常，那么异常会被记录到outcome里面。

所以，state属性是贯穿整个FutureTask最核心的属性。

isCancelled()

判断state是否>=CANCELLED，包含了CANCELLED INTERRUPTING INTERRUPTED，而这三种状态，则正好是调用cancel方法后才会出现的。

isDone()

同样的，我们一直强调，如果状态state不为NEW，那么说明任务肯定已经执行完毕了。因为任务不存在类似“任务执行中”这种状态，即便是短暂的中间态，也是发生在任务已经执行完毕，正在设置设置结果的时候。

get() 方法

看上去get方法很简单，当任务还没执行完毕或者还在设置结果是，则执行awaitDone方法来等待任务进入终止态。awaitDone的返回值是任务的状态，而不是任务的结果。任务进入终止态后，根据任务结果来返回执行结果或者抛出异常，即report(s)方法。

总结

至此，FutureTask分析的差不多了。

有这几个重要的属性：

  1. state。初始状态 ->中间状态 ->终止状态。只要不是NEW，那么表示任务已经执行完毕，没有中间状态”正在执行中“，该状态只是设置结果前使用。
  2. runner，表示当前执行任务的线程
  3. callable，表示需要执行的任务
  4. waiters，表示等待获取结果的队列，也是Treiber 栈。
  5. outcome，表示任务执行的结果，既有正常结束的结果也有抛出的异常。

使用CAS保证线程安全。

同时FutureTask也存在一些问题：

获取结果时，如果任务没有结束，那么线程会自旋或者挂起（CompletableFuture解决）；在Treiber栈中可能会在多个线程在一瞬间准备执行，会可能存在多余的无效节点，感觉会有点问题。
